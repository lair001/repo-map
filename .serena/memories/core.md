# RepoMap Core

- Local deterministic knowledge graph builder for polyglot software repos; no LLM-guessed graph edges.
- Primary source tree: `src/main/python/repomap_kg`; tests mirror under `src/test/unit/python/repomap_kg` and `src/test/int/python/repomap_kg`.
- Public specs and ADRs live under `docs/specs` and `docs/adr`; start with architecture, extractor strategy, raw observation schema, storage model, profile schema, roadmap, and the graph identity/key-vocabulary/canonicalizer ADRs.
- Raw observations are the fixture/debug/interchange boundary. See `mem:conventions` for stable-key and confidence expectations.
- Storage uses Postgres relational property-graph tables plus JSONB metadata. It retains raw observation JSON per run in `raw_observations` for replay, and now dual-writes canonical graph rows beside the current observation-derived rows. See `mem:tech_stack` and `mem:task_completion` before touching DB-backed code.
- CLI entrypoint is `repomap_kg.cli:main`; package import name is `repomap_kg`; distribution and command name are `repomap-kg`.
- Current roadmap: Phase 3 shell extractor is paused while canonicalization/storage transition work proceeds. Phase A1/A2 has graph key builders/parsers/validators plus canonical diagnostics/dataclasses/serialization in `repomap_kg.graph_keys`, `repomap_kg.canonical_diagnostics`, and `repomap_kg.canonical`. Phase B maps `file`, static `shell.command`, `shell.source`, `shell.env`, and `shell.host_mutation` raw observations into the pure canonical graph with exact golden fixture coverage for ADR 0004 current-scope cases, diagnostic placeholders, file metadata conflicts, and malformed target rebuild/placeholder behavior. Phase C raw observation retention is complete: migration `2026/06/29-001-core-create_raw_observations_table.sql` plus `load_file_observations` retain each input `RawObservation.to_dict()` with a run-local ordinal in Postgres without changing public CLI output. Phase C canonical storage dual-write is complete: migration `2026/06/29-002-core-create_canonical_graph_tables.sql` adds canonical nodes, edges, evidence, and evidence-link tables; `load_file_observations` runs the pure canonicalizer in memory and writes canonical rows alongside existing observation-derived storage. Public storage readback still uses the current `nodes`, `edges`, and `evidence` tables until a later canonical readback/compatibility slice.
